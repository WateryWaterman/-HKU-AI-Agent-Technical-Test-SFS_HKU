碰撞检测 vs 门净宽检查：更适合你这次 MVP 的是哪条路？

这份导览直接回答三件事：现在做演示与测试，应该参考哪个 IFC 版本；去哪里找可用的 IFC 样本；以及在“碰撞检测”和“门净宽检查”之间，哪条路线更稳、更容易解释、更适合 7 天内交付。

先给结论：规范参考层面优先看 IFC 4.3；但工程实现层面，样本和库兼容性不够稳时，可以用 IFC4/IFC2x3 样本先把 MVP 跑通，再把概念和属性解释对齐到 4.3。

快速判断

你想先追求最稳演示

优先门净宽 / 门属性检查，数据解释更直接，结果更容易做成人可审阅的卡片。

你想更像 BIM 工程工具

优先碰撞检测，视觉冲击更强，但几何与样本质量风险也更高。

1. IFC 版本建议

如果你问“参考哪一版规范最稳”，答案是：IFC 4.3 现在是官方最新正式标准，而且 buildingSMART 明确写了最新官方版本为 4.3.2.0，并对应 ISO 16739-1:2024。

版本策略适用场景为什么
规范阅读：IFC 4.3查实体、属性、Pset、官方术语它是最新官方标准，文档更新、门属性等查阅体验更好
样本实验：IFC4 / IFC2x3 也可接受快速拿公开模型做 parser / viewer / rule demo历史样本更多，兼容性更成熟，很多教程和 viewer 示例仍基于这些版本
最终表述：以 4.3 为口径写 README、汇报设计判断能体现你跟的是当前正式标准，而不是过时资料
也就是说，你不必教条地要求“演示样本必须全部来自 4.3”。对这次测试来说，规范认知用 4.3，工程样本用最容易跑通的版本，往往是更理性的工程判断。
一句话版本结论

4.3 是“认知和表述上的主版本”，而 IFC4 / IFC2x3 可以是“演示和兼容性上的过渡版本”。

2. IFC 样本去哪找

最推荐你先从“官方样本 + 社区经典小模型”两路并行。前者更正规，后者更容易看懂和改造。

官方首选

buildingSMART 的 Sample-Test-Files 仓库，包含多种 schema 版本和简单 PCERT discipline models，适合拿来做 parser 与 viewer 验证。

4.3 专项样本

IFC4.3.x-sample-models 更贴近 4.3，适合你在熟悉后拿来验证“我是不是按照最新规范理解对象和关系”。

经典教学模型

IfcOpenHouse 是最经典的小型教学样本之一，门、窗、墙、屋顶齐全，结构相对容易理解。

可编程复刻版

IfcOpenHouse Python 复刻 适合你一边看模型，一边看它是如何由代码生成的。

找演示样本时，优先满足这三个条件：
1) 体量小；2) 门/墙/空间等对象清晰；3) 在 viewer 中能稳定打开。不要一开始就拿超复杂综合体做首个样本。

3. 两条规则对比

维度碰撞检测门净宽检查
规则难度较高 涉及几何求交、对象过滤、容差和结果去重中等 核心是找 door 对象、读宽度、套阈值和补解释
输入依赖几何质量要求高，模型坐标、布尔表达、构件分类都影响结果更依赖门对象属性是否完整，几何要求相对没那么高
是否容易解释中等，用户会追问“这算碰撞还是容差内接触”高，用户容易理解“这扇门宽度小于阈值所以命中”
可视化效果强，红色高亮两个碰撞对象很直观中上，单门高亮 + 指标卡片也不错，但视觉冲击略弱
对属性依赖低到中，主要靠几何与对象类别高，门宽、FireExit、FireRating 等字段越完整越好
实现后扩展性高，可继续扩到墙/梁/管/设备碰撞高，可继续扩到门净高、疏散门属性、门开启方向等
演示稳定性中 易受模型质量影响较高 更容易做成稳定 demo
更适合本题 7 天 MVP适合想做“工程感 + 3D 高亮”的路线适合想做“解释性 + 稳妥交付”的路线

4. 工程风险

碰撞检测的主要坑

假阳性与假阴性、容差阈值难讲清、样本模型几何导出不稳定、结果去重和分组麻烦。如果前端还要联动高亮，多半会把时间压得更紧。

门净宽检查的主要坑

样本里不一定有标准化宽度属性；有时宽度要从类型、属性集或几何估算中找；“消防门”也不一定被显式填成 FireExit，需要加字段缺失策略。

从“可运行 + 有思考 + 有品味”的角度看，门净宽路线更容易把检查依据、计算结果、人工复核界面三件事讲完整。

5. 对你的推荐

推荐 A：最稳 MVP

做 IfcDoor 净宽检查，再附带 FireRating / FireExit 属性展示。这样既有“规则”，又有“解释”，还能体现 IFC 语义价值。

推荐 B：更炫展示

做简单 clash detection，只检查一类明确对象对，比如墙 vs 管。对象范围越窄，结果越稳。

推荐 C：双阶段

先用门规则拿下可运行底线，再补一个很轻的碰撞可视化作为加分项，而不是反过来。

如果你现在还在边学 IFC 边做原型，我更推荐你先拿门规则开局。碰撞检测可以作为第二优先级，而不是第一天就绑定在主路径上。
