# BIM 检查 MVP 指导文件

## 目的

这份文件用于指导本地 agent 与人工协作推进一个面向 HKU 技术测试的 BIM 合规检查 MVP。任务目标是在短时间内完成 1–2 条规则的可运行检查，结合可视化呈现，并体现工程判断、交互思考与结果解释能力。[cite:1]

## 总体判断

这次实现应拆成两条主路线：

- **路线 A：门净宽检查 + 可视化**，作为更稳的主线。
- **路线 B：IfcClash 接入 + 可视化**，作为几何检查路线或第二阶段增强。

门规则更容易做出“规则依据—结果—人工确认”的闭环；碰撞检测更有视觉冲击，但更依赖几何质量、对象过滤和容差解释。[cite:5][cite:1]

## 总原则

- 先把“规则内核”做好，再讨论 agent 外壳。
- 先做“可解释结果”，再做“炫展示”。
- 先保证一条规则稳定跑通，再扩第二条。
- 结果必须支持人工审核，不把自动判断伪装成最终裁决。[cite:1]

## 模块划分

不先写代码结构，只先定义模块职责与信息传递。

| 模块 | 职责 | 输入 | 输出 | 是否可并行 |
|---|---|---|---|---|
| 1. 样本与版本识别 | 识别 IFC schema、样本质量、可用对象 | IFC 文件 | schema 版本、对象概览、风险提示 | 否，最先开始 |
| 2. 规范检索与规则依据 | 查实体、Pset、属性与版本差异 | 规则主题、关键词 | 规则依据摘要、字段读取策略 | 可与 3 并行部分推进 |
| 3. 归一化层设计 | 定义内部统一字段，不直接暴露原始 schema 差异 | schema 信息、规则需求 | normalized door / clash candidate 数据 | 否，是后续核心依赖 |
| 4A. 门规则内核 | 读取门宽、属性、阈值并给出判断 | normalized door data | rule results JSON | 在 3 完成后开始 |
| 4B. Clash 内核 | 组织对象组并调用 IfcClash | normalized clash candidate data | clash results JSON | 在 3 完成后开始 |
| 5. 结果解释层 | 把原始结果转成可读证据 | 规则结果 JSON | explanation cards / audit items | 可与前端方案并行 |
| 6. 可视化适配层 | 将结果映射到 viewer 可高亮对象 | 结果 JSON、模型对象 ID | highlight payload / selection payload | 依赖 4A 或 4B |
| 7. 人工审核层 | 人确认、驳回、备注与复核 | explanation + highlight | final review notes | 最后阶段 |
| 8. 演示与叙事层 | 组织 demo 流程与提交展示 | 全部模块输出 | README、视频脚本、操作流程 | 后期收束 |

## 信息传递原则

所有模块之间传递信息时，优先传**标准化 JSON / 结构化对象**，不要直接传大段文本。这样可以减少前后层耦合，也方便 agent、后端和可视化层分别消费。

推荐的传递思路：

1. **模型输入层**输出：schema 版本、对象清单、对象 ID、属性命中情况。
2. **归一化层**输出：统一 door record / clash candidate record。
3. **规则层**输出：规则命中记录、阈值、证据字段、置信度、未确定原因。
4. **可视化层**输出：需要高亮的对象 ID、颜色、视角建议、说明文本。
5. **人工层**输出：确认/驳回/需复核、备注、最终状态。

## 路线 A：门净宽检查 + 可视化

### 目标

构建一条稳定主线：从 IFC 中找到门对象，提取宽度与相关属性，判断是否满足设定阈值，并在前端高亮对应门对象，同时显示判定依据。

### 关注重点

- 是否稳定识别 `IfcDoor`。
- 宽度字段从哪里读取，优先顺序是否明确。
- `FireRating` / `FireExit` 等属性是否存在，若缺失如何处理。
- 规则结果如何解释给人看，而不是只输出 pass/fail。

### 人工关键把握

- 决定阈值是否用于演示，还是仅演示“字段读取 + 可视化 + 风险提示”。
- 审核宽度字段来源是否可信，避免把估算值误当标准值。
- 对“字段缺失”与“不合规”做明确区分。

### 必须做到的需求

- 至少识别并列出门对象。
- 至少给出一条门规则的命中结果。
- 至少能在可视化中选中/高亮对应门。
- 至少展示判断依据与人工复核提示。

### 可并行事项

- 规范检索与样本筛选可并行。
- 结果解释模板与前端展示稿可并行。
- 宽度字段归一化完成前，不要开始最终规则判断。

## 路线 B：IfcClash 接入 + 可视化

### 目标

调用 IfcClash 完成一类对象对的碰撞检测，并把结果对象对映射到可视化层进行高亮与浏览。

### 关注重点

- 选择尽量窄的对象范围，例如墙 vs 管，避免对象过多导致噪声。
- 容差、假阳性和重复碰撞结果如何处理。
- clash result 是否能稳定映射回 viewer 中可选中的对象。

### 人工关键把握

- 先确认样本是否真的适合做 clash，不要为了“有碰撞”硬造复杂模型。
- 决定演示时是否把“接触”也当 clash，还是只展示明显穿透。
- 复核 1–2 个典型结果，确保不是解析误差。

### 必须做到的需求

- 成功调用 IfcClash 或等价流程。
- 至少输出一组 clash pair 结果。
- 至少能在可视化中高亮两个相关对象。
- 至少说明规则适用范围与误判风险。

### 可并行事项

- viewer 交互草图可先做。
- IfcClash 结果解释模板可先做。
- 对象组与容差策略未定前，不要锁定最终展示话术。

## 可视化方案讨论重点

前端具体实现可后置，但方案上现在就要定清楚以下事项：

- 使用 xeokit / web-ifc / 其他 viewer 时，如何获取对象 ID 与高亮能力。
- 是做 3D 高亮 + 侧边结果列表，还是先做列表驱动再反选模型。
- 结果中是否包含摄像机跳转、聚焦框、问题颜色语义。
- 人工审核动作是否包含“确认问题 / 标记误报 / 添加备注”。

当前更推荐：**侧边栏结果列表 + 模型高亮联动**。这样既符合审查工作流，也更容易演示“AI/自动检查只给建议，人做最后认定”。[cite:1]

## 版本探测与归一化

### 版本探测需要关注什么

- 文件 schema 是 IFC2x3、IFC4 还是 IFC4.3。
- 当前规则依赖字段在该版本中是否稳定存在。
- 当前样本里对象与属性是“标准具备”还是“样本实际填充”。

### 归一化层为什么重要

不同版本、不同导出器和不同样本的字段位置可能并不完全一致。buildingSMART 对兼容性的讨论也区分了文件级兼容与服务级转换兼容，因此业务逻辑不应直接耦合单一版本原始字段。[cite:63]

### 归一化层需要做到什么

- 统一 door record：对象 ID、名称、楼层、宽度、宽度来源、消防相关属性、缺失说明。
- 统一 clash record：对象 A、对象 B、对象类别、碰撞类型、证据、容差、解释说明。
- 明确“字段不存在”“字段未填”“字段可估算”这三种不同状态。

## 人工与 agent 的分工

### 适合交给本地 agent 的事

- 本地规范检索与索引维护。
- 样本扫描、对象统计、属性命中情况汇总。
- 规则依据文档整理。
- 结果 JSON 的转换、清洗与摘要生成。

### 必须由人工把握的事

- 规则阈值是否合理。
- 哪些结果能作为演示主案例。
- 哪些样本质量太差，不值得继续投入。
- 是否把某条结果表述为“问题”还是“待人工复核”。
- 演示重点放在解释性还是炫技性。

## 并行与依赖

### 可以并行

- 样本筛选 与 规范索引建立。
- 门规则研究 与 clash 方案预研。
- 结果解释模板 与 viewer 交互草图。

### 不可并行或不宜过早并行

- 归一化层未定前，不宜大规模做规则实现。
- 结果 JSON 未定前，不宜把前端交互写死。
- 样本未确认前，不宜录制最终演示脚本。

## 执行顺序建议

1. 确认 2–3 个小样本可打开、可解析。
2. 完成本地规范索引第一版。
3. 完成版本探测与归一化字段草案。
4. 先打通门净宽检查。
5. 补结果解释与可视化联动。
6. 视时间决定是否接入 IfcClash。
7. 收束 README、prompts、demo 脚本与视频。

## 交给 opencode 的任务说明

1. 按这份文件拆解阶段任务，不要一次性铺满全部实现。
2. 先围绕路线 A 建立最小闭环，再评估路线 B。
3. 每完成一个阶段，都输出：已确认事实、未确认事项、下一阶段依赖。
4. 任何版本兼容结论必须标注证据来源，不允许凭名称相似直接判断。
5. 所有结果输出都必须保留“人工复核入口”，不把自动结果写成最终结论。
