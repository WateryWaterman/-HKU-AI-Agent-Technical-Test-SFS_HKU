1. IFC 与 BIM

BIM 更像一种“建筑信息建模的方法与工作方式”，而 IFC 则是其中最核心的开放交换标准之一。buildingSMART 将 IFC 定义为对建筑与基础设施的标准化数字描述，它不仅能表达对象几何，还能表达身份、属性、关系、过程和参与方信息。

BIM 像什么

是把建筑当成“可计算的信息系统”来组织，而不是只画图。模型里不只有形状，还有构件类型、空间关系、材料、设备、属性与生命周期信息。

IFC 像什么

是 openBIM 的公共语言。它让 Revit、Archicad、Tekla、浏览器工具、分析工具之间，可以尽量用同一种结构理解“墙、门、空间、属性、关系”。

对你最重要的认识：做规则检查时，你往往不是在“看一个 3D 模型”，而是在“读一组有几何 + 语义 + 关系的数据对象”。

2. 它解决的问题

IFC 诞生的核心原因，是建筑行业长期存在软件割裂、数据孤岛和厂商锁定问题。buildingSMART 明确把 IFC 定义为 vendor-neutral 的国际开放标准，用来在不同软件、硬件和接口之间交换与保存建筑信息。

互操作

不同软件作者、审图方、施工方、业主方都可能使用不同系统，IFC 提供共同语义底座。

长期保存

项目交付后仍需保留可读信息，IFC 适合作为归档或 as-built 信息载体。

自动化

既然属性和关系是机器可解释的，就能做规则检查、算量、质检、问答和协作流转。

也正因为 IFC 是“共识层”，它不会替你自动完成所有专业判断；很多检查能力仍然要靠外部代码、规则库或业务逻辑来实现。

3. 与类似格式区别

对象更像什么强项局限
IFC开放 BIM 交换标准几何 + 语义 + 属性 + 关系，适合规则检查与跨软件交换结构复杂，解析门槛高，模型质量受导出质量影响
DWG / CAD 图纸绘图文件2D 表达成熟，存量巨大语义往往弱，很多检查需要二次识别或人工约定
Revit 原生文件作者软件内部模型信息通常更完整，原生编辑能力强不开放，生态依赖特定软件
glTF / OBJ偏几何渲染格式Web 渲染友好通常缺少 IFC 这种工程语义和关系网络
简化 JSON你为任务定制的中间表示最适合做 MVP、规则验证与调试需要你自己定义语义规则，通用性弱
一个重要理解：IFC 不等于“.ifc 文本文件”

buildingSMART 说明 IFC 数据可以编码为 STEP、XML、JSON 等多种形式。也就是说，IFC 的核心先是“schema / data model”，其次才是具体落地文件编码。

4. 碰撞检测怎么来

IFC 本身提供的是对象、几何与关系的标准化表达，但“碰撞检测”不是 IFC 文件天然自带的一键能力，而通常要由外部工具或代码基于模型几何去计算。IfcOpenShell 文档中的 IfcClash 就是一个明确例子，它既可作为 CLI，也可作为库，对一个或多个 IFC 模型执行 clash detection。

IFC 提供什么

对象类别、几何表示、空间层级、属性集、对象关系。这些是“检查的原材料”。

代码提供什么

对象筛选、几何求交、容差判断、规则阈值、结果分组、可视化与报告导出。这些才是“检查器本体”。

所以你做“碰撞检测 MVP”时，可以把它理解成：
读取 IFC → 提取候选对象（如墙/梁/管道）→ 做碰撞计算 → 输出命中对 → 在前端高亮与解释。

5. 门与消防属性

IFC 里的门并不是一团匿名几何，标准中有 IfcDoor 以及对应的常见属性集 Pset_DoorCommon。IFC 4.3 文档显示，Pset_DoorCommon 中就包含 FireRating 与 FireExit 等属性，这意味着“消防门/疏散门相关语义”在标准层面是有位置的。

标准里可能已有

FireRating、FireExit、IsExternal、SelfClosing、SmokeStop 等字段，适合直接作为规则输入。

现实里仍要判断

模型未必填了这些属性，或者不同软件导出不统一，所以你仍然需要“字段存在性检查 + 缺省值策略 + 业务判断层”。

关键点不是“IFC 里有没有消防门”，而是“样本模型是否真的把 FireRating / FireExit 等属性正确填出来”。标准能表达，不等于数据一定完整。
这对你的规则设计意味着什么

若你做“门净宽 + 消防门属性完整性”这类规则，会比“从纯 CAD 图识别哪扇门是消防门”简单很多。前者依赖 IFC 的显式语义，后者往往需要额外识别逻辑。

6. 工具链理解

工具你可以把它理解成适合放在哪一层
Three.js通用 3D 渲染引擎最底层显示层，灵活，但你要自己搭 BIM 语义、选中、高亮等逻辑
web-ifc浏览器内 IFC 解析器前端直接读取 IFC，适合做轻量解析、对象查询、浏览器端实验
xeokit偏 BIM/AEC 的 Web 可视化框架展示层更强，支持 IFC 相关加载与语义交互，适合做“检查结果可视化”
IfcOpenShell / IfcClashPython 侧 IFC 解析与工程处理库后端规则层，适合属性读取、对象筛选、碰撞检测、报告生成
对你这题最自然的分工通常是：IfcOpenShell 做检查，xeokit 或 web-ifc 做展示。

8. 对你的 MVP 启发

如果你选碰撞检测

优先用 IfcOpenShell / IfcClash 在后端算结果，前端只负责把 clash pair 列出来并高亮对象。这样“规则正确性”和“展示复杂度”是可分离的。

如果你选门规则

优先检查 IfcDoor 的宽度、属性完整性和 FireExit / FireRating。它比纯几何碰撞更容易做出“解释为什么命中”的结果卡片。
